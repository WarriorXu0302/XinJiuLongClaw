# 新鑫久隆 ERP 系统架构书

> 版本：v2.0 | 日期：2026-04-20

---

## 1. 系统概述

### 1.1 定位

面向多品牌白酒经销业务的企业资源管理系统。一个公司下管理多个品牌事业部（青花郎 / 五粮液 / 汾酒 / 珍十五等），每个品牌独立核算，统一管理。

### 1.2 核心理念

- **品牌 = 事业部**：每品牌有独立的账户（现金/F类/融资）、独立的库存、独立的薪资模板、独立的利润核算
- **收款统一入口**：所有客户回款进公司总资金池（master），再按需调拨到品牌
- **数据库层安全**：PostgreSQL RLS 行级安全策略，防止上层（含 AI Agent）绕过权限

### 1.3 规模

| 指标 | 数量 |
|---|---|
| 后端 API 端点 | 267 个 |
| 数据库表 | 61 张 |
| 前端页面 | 60 个 |
| 用户角色 | 9 种 |
| RLS 保护表 | 14 张 |
| 利润台账科目 | 11 项 |


### 2.3 数据库连接双引擎

| 引擎 | 角色 | 用途 | RLS |
|---|---|---|---|
| `app_engine` | erp_app | FastAPI 请求处理 | **受约束** |
| `admin_engine` | erpuser | Alembic / seed / 后台任务 | 绕过 |

每个 API 请求在事务开始时通过 `SET LOCAL` 注入用户上下文（brand_ids / is_admin / employee_id），PostgreSQL RLS 策略自动过滤数据行。

### 2.4 服务端分页

所有列表类接口统一返回分页格式：

```json
{ "items": [...], "total": 总数 }
```

支持 `skip`（偏移量，默认 0）和 `limit`（每页条数，默认 50）查询参数。前端使用 `extractItems()` 工具函数兼容提取数组数据。

## 4. 资金流体系

### 4.1 账户结构

```
公司总资金池（master cash）
  │
  ├── 青花郎
  │     ├── 现金账户（日常支出：工资/采购/稽查/兑付）
  │     ├── F类账户（专款订货：厂家政策到账）
  │     ├── 融资账户（厂家融资）
  │     └── 回款账户（统计虚账：记录付给厂家的货款）
  │
  ├── 五粮液（同上）
  ├── 汾酒（同上）
  └── 珍十五（同上）
```

### 4.2 资金流向规则

| 动作 | 资金流向 |
|---|---|
| **客户回款** | → master 现金（公司总资金池） |
| **政策/F类到账** | → 品牌 F 类账户 |
| **工资补贴到账** | → 品牌现金账户 |
| **工资发放** | 品牌现金 → 员工 |
| **采购付款** | 品牌现金/F类/融资 → 厂家 |
| **稽查回收** | 品牌现金 → 对方（余额不足报错） |
| **报销付款** | 品牌现金 → 员工 |
| **垫付返还** | 品牌现金 → 业务员/客户 |
| **资金调拨** | master → 品牌现金/融资（需老板审批） |

### 4.3 三种结算模式

| 模式 | 客户付 | 补差方 | 公司应收 | 提成基数 | 全款触发 |
|---|---|---|---|---|---|
| customer_pay | 指导价全额 | 无 | total_amount | total_amount | 客户付齐 |
| employee_pay | 到手价 | 业务员补差 | total_amount | total_amount | 客户+业务员凑齐 |
| company_pay | 到手价 | 公司让利 | deal_amount | deal_amount | 客户付到手价即可 |

### 4.4 收款→确认→兑付闭环

```
业务员分批上传凭证（每笔建 Receipt + 进 master + 更新 payment_status）
    ↓
payment_status = fully_paid → 页面锁定 + 通知财务
    ↓
审批中心 → 财务确认收款 → order.status = completed
    ↓
政策兑付解锁（fulfilled → 自动生成垫付返还 FinancePaymentRequest）
    ↓
厂家到账 → F类账户 / 品牌现金（工资补贴）
    ↓
垫付返还 → 老板审批 → 品牌现金扣款 → 打给业务员/客户
```

## 5. 权限体系

### 5.1 角色矩阵

| 角色 | 总资金池 | 订单 | 工资 | 财务 | 库存 | 稽查 | 审批 | 用户管理 | 商品管理 |
|---|---|---|---|---|---|---|---|---|---|
| admin | ✅ | 全部 | 全部 | 全部 | 全部 | 全部 | 全部 | 全部 | 全部 |
| boss | ✅ | 全部 | 全部 | 全部 | 全部 | 全部 | 全部 | 全部 | 创建/编辑/删除 |
| finance | ❌隐藏 | 全品牌 | ❌ | 全部 | 只读 | 只读 | 收款/报销 | ❌ | ❌ |
| sales_manager | ❌ | 本品牌 | ❌ | ❌ | ❌ | ❌ | 目标 | ❌ | ❌ |
| salesman | ❌ | 仅自己 | 仅自己 | ❌ | ❌ | ❌ | ❌ | ❌ | ❌ |
| hr | ❌ | ❌ | 全部 | ❌ | ❌ | ❌ | 请假 | 查看 | ❌ |
| warehouse | ❌ | 出库单 | ❌ | ❌ | 授权仓 | ❌ | ❌ | ❌ | 创建/编辑 |
| purchase | ❌ | ❌ | ❌ | ❌ | 只读 | ❌ | ❌ | ❌ | ❌ |

> **用户管理权限说明**：创建用户、编辑用户、重置密码、角色分配均要求 boss 角色。hr 仅可查看用户列表。
> **商品管理权限说明**：商品创建/编辑要求 boss 或 warehouse 角色；商品和品牌的删除要求 boss 角色。所有 DELETE 端点均有角色校验。

### 5.2 数据隔离层次

```
第 1 层：PostgreSQL RLS（14 张表强制品牌隔离）
  → 即使 raw SQL / Agent prompt 注入也无法越权
  → SET LOCAL app.brand_ids 在事务内生效

第 2 层：应用层 helper（permissions.py）
  → require_role / can_see_salary / can_see_master_account
  → API 入口直接 403，不打到数据库

第 3 层：前端菜单过滤 + AuthGuard
  → 按角色隐藏菜单项
  → 路由级 requiredRoles 拦截
```

## 7. 薪资体系

### 7.1 底薪来源

```
BrandSalaryScheme（品牌 × 岗位）
  ├── fixed_salary        固定底薪
  ├── variable_salary_max  浮动底薪上限（考核满分值）
  ├── attendance_bonus_full 全勤奖全额
  ├── commission_rate      销售提成率
  └── manager_share_rate   管理提成率
```

员工入职归属某品牌（主属品牌），底薪按该品牌×岗位的模板取。

### 7.2 工资生成规则

| 项目 | 计算方式 |
|---|---|
| 固定底薪 | BrandSalaryScheme.fixed_salary |
| 浮动底薪 | 考核项 earned_amount 合计 |
| 销售提成 | Σ(订单回款基数 × 品牌提成率 × KPI 系数) |
| 管理提成 | 下属业务员回款 × manager_share_rate |
| 全勤奖 | 全额 × 请假梯度（0天100% / 1天80% / ... / ≥5天0%；迟到=0） |
| 达标奖 | 按销售目标完成率阶梯（100%档 / 120%档） |
| 社保代扣 | Employee.social_security（个人部分） |
| 厂家补贴 | **不进工资条**（独立走政策应收） |

### 7.3 工资审批流

```
draft → pending_approval → approved → paid
  ↓                          ↑
rejected ──────────────────┘
```

## 8. 利润台账

### 8.1 科目体系

| # | 科目 | 方向 | 数据来源 |
|---|---|---|---|
| 1 | 订单销售利润 | 收入 | (售价-成本)×数量，company_pay 用到手价 |
| 2 | 政策兑付盈利 | 收入 | PolicyRequestItem.profit_loss > 0 |
| 3 | 稽查清理盈利 | 收入 | 已执行案件 profit_loss > 0 |
| 4 | F类到账差额 | 收入 | arrival_amount - actual_cost |
| 5 | 回款返利 | 收入 | 年底手动（预留） |
| 6 | 报销费用 | 支出 | Expense.status=paid |
| 7 | 政策兑付亏损 | 支出 | PolicyRequestItem.profit_loss < 0 |
| 8 | 稽查外流亏损 | 支出 | 已执行案件 profit_loss < 0 |
| 9 | 融资利息 | 支出 | FinancingRepayment.interest_amount |
| 10 | 分货差价 | 支出 | ExpenseClaim(share_out)（预留） |
| 11 | 人力成本净额 | 支出 | 工资实发+公司社保-厂家补贴到账 |

### 8.2 品牌独立核算

利润台账支持按品牌筛选——每个品牌的收入、支出、净利润独立计算。稽查盈亏只算已执行案件（按执行时间过滤），未执行的预估不污染利润。

## 9. 稽查体系

### 9.1 五种案件类型

| 代码 | 方向 | 说明 | 盈亏公式 |
|---|---|---|---|
| A1 outflow_malicious | 外流 | 恶意窜货→回收→备用库 | -(回收价-到手价)×瓶数-罚款 |
| A2 outflow_nonmalicious | 外流 | 非恶意→回收→主仓 | (指导价-回收价)×瓶数-罚款 |
| A3 outflow_transfer | 外流 | 被转码 | -罚款 |
| B1 inflow_resell | 流入 | 加价回售 | (回售价-买入价)×瓶数+奖励 |
| B2 inflow_transfer | 流入 | 转码入库 | (指导价-买入价)×瓶数+奖励 |

### 9.2 资金规则

- 所有扣款从**品牌现金账户**（不是总资金池）
- 扣款前校验余额，不足则报错提示调拨
- 执行后 status=executed + 记 closed_at → 才进利润台账

## 10. 库存体系

### 10.1 双轨制

- **数量账**：`Inventory`（仓库×商品×批次 聚合数量）
- **条码账**：`InventoryBarcode`（单瓶条码 + 状态 IN_STOCK / OUTBOUND / LOCKED / INVALID）

### 10.2 成本溯源

`StockOutAllocation` 记录每张订单的出库成本分配：
- 优先精确条码匹配（BARCODE_EXACT）
- 兜底 FIFO 先进先出（FIFO_FALLBACK）

### 10.3 仓库类型

| 类型 | 用途 |
|---|---|
| main | 主仓（正常销售出库） |
| backup | 备用仓（稽查回收入库） |
| tasting | 品鉴物料仓（政策兑付出库） |
| retail | 零售仓 |
| wholesale | 批发仓 |
